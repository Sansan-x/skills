# Go代码安全审计方法论

## 方法论概述

本审计方法论结合了白盒代码审计、自动化安全扫描和基于知识库的模式匹配三种方法，形成分层递进的审计策略。

## 1. 审计方法分层

### 1.1 第一层：自动化工具扫描

**目的**：快速发现已知的常见漏洞模式，建立审计基线。

**工具链**：

| 工具 | 检测能力 | 误报率 | 推荐场景 |
|------|---------|--------|---------|
| govulncheck | 已知CVE的依赖漏洞 | 极低 | 所有项目 |
| gosec | OWASP Top 10相关Go漏洞 | 中 | 所有项目 |
| staticcheck | 代码质量和潜在bug | 低 | 所有项目 |
| go vet | 可疑代码结构 | 低 | 所有项目 |
| go test -race | 数据竞态检测 | 极低（需运行测试） | 有测试的项目 |
| errcheck | 未检查的error返回值 | 低 | 所有项目 |
| nilaway | nil解引用检测 | 中 | 所有项目 |

**执行顺序**：
1. `govulncheck` — 最快速，检查依赖漏洞
2. `go vet` + `staticcheck` — 编译级检查
3. `gosec` — 安全规则扫描
4. 其他专项工具

### 1.2 第二层：模式匹配审计

**目的**：基于go-vuln-lib攻击模式库，发现已知攻击模式的实例。

**方法**：
1. 使用grep_patterns进行初步代码搜索
2. 对匹配位置进行AST级结构比对
3. 验证preconditions是否满足
4. 应用上下文过滤减少误报

**特点**：
- 可检测工具不覆盖的Go特有安全模式
- 支持5GC特定模式
- 随模式库更新持续增强

### 1.3 第三层：数据流分析

**目的**：发现自动化工具无法检测的深层漏洞——从不可信输入到敏感操作的未净化数据流。

**方法**：
1. 枚举所有外部输入点（Source）
2. 枚举所有敏感操作点（Sink）
3. 追踪每条Source→Sink路径上的数据流
4. 检查路径上是否有有效净化

**适用于**：
- 注入类漏洞（SQL/命令/模板注入）
- SSRF
- 路径遍历
- 信息泄露

### 1.4 第四层：逻辑审计

**目的**：发现业务逻辑层面的安全缺陷。

**方法**：
- 理解业务逻辑和安全需求
- 检查认证、授权、会话管理的正确性
- 验证安全假设是否成立
- 5GC：验证3GPP安全规范合规性

**适用于**：
- 认证绕过
- 权限提升
- 业务流程违规
- 协议实现缺陷

## 2. 审计流程详细规范

### 2.1 预审计准备

```
1. 获取代码：
   - 确定审计的代码分支和版本
   - git clone并checkout到目标版本
   - 确认代码可以编译通过：go build ./...

2. 环境准备：
   - 安装Go工具链（与项目go.mod中的版本匹配）
   - 安装审计工具（govulncheck, gosec, staticcheck）
   - 准备go-vuln-lib攻击模式库

3. 范围确认：
   - 明确审计范围（全项目 or 特定包）
   - 是否包含第三方依赖的审计
   - 是否包含配置文件审计
   - 是否需要5GC合规性审计
```

### 2.2 信息收集阶段

```
1. 代码统计：
   - 总行数、包数量、文件数量
   - 外部依赖数量及版本
   - 测试覆盖率

2. 架构分析：
   - 绘制组件依赖图
   - 标识外部接口和内部边界
   - 识别安全关键组件

3. 历史分析：
   - 查看过去的安全修复
   - 检查是否有安全相关的Issue
   - 分析代码变更频率高的安全相关文件
```

### 2.3 漏洞挖掘阶段

按照四层方法从浅入深执行。每层的发现都要记录到统一的候选漏洞列表中。

**关键原则**：
- 宁可多报后验证，不要漏报
- 每个发现都要记录证据
- 注意漏洞间的关联性

### 2.4 验证与评级阶段

对候选漏洞逐一验证：

```
验证清单:
1. 代码可达性：漏洞代码是否有实际的调用路径？
2. 输入可控性：攻击者能否控制到达漏洞点的输入？
3. 净化有效性：是否存在遗漏的净化操作？
4. 环境因素：部署环境是否提供了额外防护？
5. 影响确认：成功利用后的实际影响是什么？
```

### 2.5 报告阶段

按照SKILL.md Phase 10定义的格式生成报告。

## 3. Go语言审计特殊考量

### 3.1 Go编译模型的影响

- Go是静态编译语言，不存在动态加载漏洞（除plugin包外）
- 反射（reflect）增加了代码路径的复杂性
- 接口的隐式实现使得调用链分析更复杂
- Go的link-time dead code elimination意味着某些代码虽然存在但不会编译进二进制

### 3.2 Go标准库的安全特性

**已加固的领域**：
- `crypto/tls`: 默认安全的TLS配置（TLS 1.2+，强密码套件）
- `net/http`: 请求大小限制、超时支持
- `html/template`: 自动上下文敏感转义
- `database/sql`: 原生支持参数化查询

**仍需注意的领域**：
- `text/template`: 不做HTML转义
- `encoding/xml`: 支持外部实体（需手动禁用）
- `net/http` 默认client跟随重定向
- `os/exec`: 需正确使用以避免注入

### 3.3 Go模块依赖审计

```bash
# 列出所有直接和间接依赖
go list -m all

# 检查依赖是否有已知漏洞
govulncheck ./...

# 检查是否使用了replace指令（可能引入非官方代码）
grep "replace" go.mod

# 检查是否有retract指令（被撤回的版本）
grep "retract" go.mod

# 分析依赖树中的关键安全库
go mod graph | grep -E "crypto|auth|jwt|oauth|tls"
```

## 4. 审计效率优化

### 4.1 高回报审计区域

按照漏洞发现概率排序的审计重点区域：

1. **HTTP处理器**：直接处理用户输入，漏洞密度最高
2. **认证/授权中间件**：逻辑错误影响全局
3. **数据库操作**：SQL注入仍是常见问题
4. **外部服务调用**：SSRF和信息泄露
5. **文件操作**：路径遍历
6. **命令执行**：命令注入
7. **并发代码**：数据竞态
8. **错误处理**：被忽略的安全关键错误

### 4.2 快速定位技巧

```bash
# 找到最可能存在漏洞的文件
# 大文件中的处理函数
find . -name "*.go" -not -path "./vendor/*" | xargs wc -l | sort -rn | head -20

# 有大量外部输入处理的文件
grep -rl "http\.Request\|grpc\.\|net\.Conn" --include="*.go" | sort | uniq -c | sort -rn

# 最近修改频繁的安全相关文件
git log --since="6 months ago" --name-only --pretty=format: -- "*.go" | sort | uniq -c | sort -rn | head -20
```

### 4.3 审计时间分配建议

| 阶段 | 占比 | 说明 |
|------|------|------|
| 项目理解 | 15% | 不可跳过，直接影响后续效率 |
| 自动化扫描 | 10% | 工具运行+结果整理 |
| 模式匹配 | 20% | 基于攻击模式库的系统化搜索 |
| 数据流分析 | 25% | 最耗时但回报最高 |
| 业务逻辑审计 | 15% | 需要深入理解业务 |
| 验证与评级 | 10% | 确认并评估漏洞 |
| 报告撰写 | 5% | 结构化输出 |

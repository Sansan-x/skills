# 审计报告模板

在阶段8中使用此模板构建最终安全审计报告。将各前序阶段的发现填入对应章节。

---

## 报告输出规范

**输出目录：** `./reports/`
**文件命名：** `[项目名称]-goaudit-[YYYYMMDD-HHMM].md`
**示例：** `./reports/myproject-goaudit-20260319-1430.md`

在生成报告前，先创建输出目录：
```bash
mkdir -p ./reports
```

项目名称获取规则：
1. 优先从 `go.mod` 的 module 声明中提取最后一个路径段
2. 如果不可用，使用项目根目录名
3. 移除名称中的特殊字符，仅保留字母、数字、连字符和下划线

时间使用审计执行时的当前时间，格式为 `YYYYMMDD-HHMM`。

```markdown
# Go代码安全审计报告：[项目名称]

**报告版本：** 1.0
```

## 第5章：详细发现

详细发现有两种输出方式，根据漏洞数量选择：

### 方式A：内联输出（优先使用）

当篇幅允许完整输出所有漏洞时，在主报告中按等级排列所有漏洞（致命 → 严重 → 一般 → 提示），每个漏洞包含完整的6个必填子章节。

### 方式B：分文件输出（篇幅受限时）

当漏洞数量较多无法在单份报告中完整输出时，将详细发现按等级拆分为独立文件，输出到 `./reports/` 目录。主报告第5章仅保留指引说明。

**分文件命名：** `VULN-[起始ID]-[结束ID]_[等级].md`

**分文件结构：** 每个文件标题为 `# 详细发现 VULN-[起始ID]-[结束ID]_[等级]`，内容为该等级下所有漏洞的完整详细发现（使用与下方完全相同的单漏洞模板）。

**方式B时主报告第5章写法：**

```markdown
## 5. 详细发现

详细发现已按漏洞等级拆分为独立文件，请查阅 `./reports/` 目录下的以下文件：

| 文件 | 等级 | 漏洞数量 |
|------|------|----------|
| [VULN-001-003_致命.md](./VULN-001-003_致命.md) | 致命 | 3 |
| [VULN-004-008_严重.md](./VULN-004-008_严重.md) | 严重 | 5 |
| [VULN-009-015_一般.md](./VULN-009-015_一般.md) | 一般 | 7 |
| [VULN-016-018_提示.md](./VULN-016-018_提示.md) | 提示 | 3 |
```

## 格式指南

1. **严重性标识** — 使用中文标签：严重、高危、中危、低危、信息

2. **发现完整性（最高优先级规则）** — 所有漏洞必须被完整输出，一个不少。方式A：全部写入主报告第5章；方式B：按等级拆分为4个文件（`VULN-[起始]-[结束]_致命.md` 等）。无论哪种方式，最终通过"详细发现完整性校验"表格确认总数一致。绝不允许用"以此类推"省略漏洞

3. **每个发现的6个必填子章节** — 对每个漏洞，以下6个子章节缺一不可：
   - 5.[N].1 漏洞描述：问题是什么、为什么危险、影响是什么
   - 5.[N].2 漏洞代码：包含完整函数体，标注文件路径、行号和漏洞行
   - 5.[N].3 数据流路径：结构化表格格式的 Source → 传播 → Sink 追踪
   - 5.[N].4 利用场景：逐步攻击描述 + 可直接复现的PoC（curl/代码/脚本）
   - 5.[N].5 影响：机密性、完整性、可用性三维评估
   - 5.[N].6 修复建议：修复前（漏洞代码）与修复后（安全代码）的完整对比

4. **代码片段** — 始终包含文件路径和行号以便追溯。漏洞代码片段应包含完整函数体或足够上下文

5. **数据流路径** — 使用结构化表格格式（如第5章模板所示），明确标注：
   - Source：污点源位置、类型、表达式
   - 传播路径：逐节点列出位置、操作、污点状态
   - Sink：汇聚点位置、危险操作、表达式
   - 净化检查：是否存在净化、有效性评估
   - 结论：可达性、可利用性、完整调用链

6. **利用PoC** — 每个发现的利用场景必须包含可直接复现的PoC，不能仅用文字描述。PoC形式包括但不限于：curl命令、HTTP原始请求、Python/Go利用脚本、gRPC调用示例

7. **修复代码对比** — 每个发现必须同时展示"修复前"（漏洞代码）和"修复后"（安全代码）的完整对比，附修复原因说明

8. **发现ID** — 使用顺序 `VULN-001`、`VULN-002` 格式

9. **攻击链** — 使用编号步骤和清晰过渡；每步必须引用具体漏洞ID

10. **报告语言** — 全文使用中文，技术术语保持英文（如CWE、CVSS、SSRF等）

11. **报告长度** — 详尽但不冗余。每个发现应自包含，开发者仅凭报告即可修复

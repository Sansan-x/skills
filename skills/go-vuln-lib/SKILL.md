# Go 攻击模式库构建

系统化收集、提取和管理Go语言攻击模式的技能。通过从多种安全信息来源中提取结构化攻击模式，构建可复用的Go安全测试知识库。

## 工作流概览

攻击模式库构建按 5 个阶段执行，支持从单一来源或多来源组合收集。

```
阶段1: 来源识别与准备
    ↓
阶段2: 模式提取
    ↓
阶段3: 结构化与标准化
    ↓
阶段4: 质量验证与去重
    ↓
阶段5: 库输出与索引
```

---

## 阶段1：来源识别与准备

确定攻击模式的收集来源并准备输入材料。支持四种来源，可单独使用也可组合。

### 1.1 来源类型

| 来源 | 说明 | 输入形式 |
|------|------|----------|
| **go-vuln-insight报告** | 从开源Go项目安全问题洞察到的攻击方法 | go-vuln-insight skill输出的Markdown报告文件 |
| **go-codehub-issue报告** | 从codehub项目issue中提取的Go安全问题 | go-codehub-issue skill输出的Markdown报告文件 |
| **安全测试指导文档** | Go语言安全测试方法论和指南 | 用户提供的Markdown文档 |
| **专家漏洞案例** | 安全研究员编写的真实漏洞分析 | 用户提供的Markdown文档或代码文件 |

### 1.2 来源准备

对每种来源类型执行预处理：

**go-vuln-insight报告：**
- 定位报告文件（通常位于 `reports/` 目录下，命名含 `vuln-insight`）
- 识别报告中的漏洞发现章节、攻击方法描述、PoC代码
- 提取报告元数据（目标项目、分析日期、漏洞统计）

**go-codehub-issue报告：**
- 定位报告文件（通常位于 `reports/` 目录下，命名含 `codehub-issue`）
- 识别报告中的安全问题分类、影响分析、复现步骤
- 提取报告元数据（项目来源、issue编号、安全标签）

**安全测试指导文档：**
- 扫描文档目录结构
- 识别测试方法章节、漏洞类型覆盖、代码示例
- 建立文档主题索引

**专家漏洞案例：**
- 识别案例结构（漏洞背景、触发条件、利用链、修复方案）
- 提取嵌入的代码示例
- 标注案例的真实程度和严重性

在进入阶段2前，输出来源清单和预处理摘要。

---

## 阶段2：模式提取

从每种来源中系统性提取攻击模式要素。

### 2.1 从go-vuln-insight报告提取

报告中重点关注：

1. **漏洞发现章节** — 提取每个漏洞的：
   - 漏洞类型和CWE编号
   - 触发条件和前置条件
   - 涉及的Go API或框架特性
   - 报告中的漏洞代码片段

2. **数据流分析** — 提取：
   - Source（污点源）→ Sink（汇聚点）路径
   - 传播中涉及的中间函数
   - 净化措施的有无和有效性

3. **攻击链分析** — 提取：
   - 单步攻击和组合攻击
   - 攻击前置条件
   - 可达的最终影响

### 2.2 从go-codehub-issue报告提取

报告中重点关注：

1. **安全问题** — 提取：
   - 问题分类（漏洞类型）
   - 影响范围和严重程度
   - 问题触发条件

2. **复现信息** — 提取：
   - 复现步骤
   - 最小复现代码
   - 环境要求

3. **修复方案** — 提取：
   - 补丁代码或修复建议
   - 修复前后的代码对比

### 2.3 从安全测试指导文档提取

文档中重点关注：

1. **测试方法论** — 提取：
   - 测试策略和检测手段
   - 工具使用方法
   - 自动化测试脚本

2. **漏洞模式说明** — 提取：
   - 漏洞原理解释
   - 典型代码模式
   - 常见误用场景

3. **安全编码指南** — 提取：
   - 安全替代方案
   - 最佳实践代码示例
   - 防御性编程技巧

### 2.4 从专家漏洞案例提取

案例中重点关注：

1. **漏洞背景** — 提取：
   - 真实项目信息（已脱敏）
   - 漏洞发现经过
   - 影响评估

2. **技术细节** — 提取：
   - 完整的漏洞利用链
   - 关键代码路径
   - 环境和配置依赖

3. **修复与防御** — 提取：
   - 实际修复方案
   - 纵深防御措施
   - 经验教训总结

## 阶段3：结构化与标准化

将提取的原始模式信息转化为标准的攻击模式条目。加载攻击模式结构化存储规范：[references/attack-pattern-schema.md](./references/attack-pattern-schema.md)

### 3.1 攻击模式条目结构

每个攻击模式条目包含以下核心字段：

```yaml
pattern_id: "GO-ATK-[类别缩写]-[序号]"
name: "模式名称"
category: "漏洞类别"
severity: "严重 | 高危 | 中危 | 低危"
cwe_ids: ["CWE-XXX"]
source_type: "vuln-insight | codehub-issue | security-guide | expert-case"
source_ref: "来源引用"

description:
  summary: "一句话概要"
  background: "漏洞背景与原理"
  impact: "安全影响"
  preconditions: ["前置条件列表"]

test_method:
  strategy: "测试策略描述"
  detection_points: ["检测点列表"]
  test_steps: ["测试步骤"]
  tools: ["适用工具"]
  automation_hint: "自动化提示"

vuln_pattern:
  code_example: "包含漏洞的典型Go代码"
  sink_functions: ["危险函数列表"]
  source_types: ["污点源类型"]
  dataflow: "source → sink 数据流描述"
  trigger_conditions: ["触发条件"]

safe_pattern:
  code_example: "安全的Go代码实现"
  fix_strategy: "修复策略描述"
  defense_layers: ["纵深防御措施"]

metadata:
  go_versions: ["适用Go版本"]
  frameworks: ["相关框架"]
  tags: ["标签列表"]
  created_at: "创建时间"
  updated_at: "更新时间"
  confidence: "高 | 中 | 低"
```

### 3.2 条目填充规则

对每个提取的模式，按以下优先级填充字段：

1. **必填字段** — `pattern_id`、`name`、`category`、`severity`、`description.summary`、`vuln_pattern.code_example`、`safe_pattern.code_example`
2. **推荐填充** — `cwe_ids`、`test_method`、`vuln_pattern.sink_functions`、`vuln_pattern.dataflow`
3. **可选字段** — `metadata`、其余字段

来源不同，可填充的字段侧重不同：

| 来源 | 强项字段 | 可能缺失 |
|------|----------|----------|
| go-vuln-insight | dataflow、sink_functions、trigger_conditions | test_method.automation_hint |
| go-codehub-issue | code_example、test_steps、fix_strategy | dataflow完整路径 |
| 安全测试指导 | test_method、detection_points、tools | 真实code_example |
| 专家漏洞案例 | 全面（最完整来源） | go_versions（可能过时） |

缺失字段可标记为 `"[待补充]"`，在后续迭代中由其他来源补全。

### 3.3 类别编码

攻击模式按漏洞类别编码：

| 类别缩写 | 类别名称 | 说明 |
|----------|----------|------|
| SQLI | SQL注入 | SQL查询构造中的注入漏洞 |
| CMDI | 命令注入 | OS命令执行中的注入漏洞 |
| XSS | 跨站脚本 | HTML/JS输出中的注入漏洞 |
| SSRF | 服务端请求伪造 | 不受控的出站请求 |
| PTR | 路径穿越 | 文件路径操控漏洞 |
| AUTH | 认证缺陷 | 认证机制绕过或弱化 |
| IDOR | 访问控制缺陷 | 不当的对象级授权 |
| CRYPTO | 密码学失败 | 加密和随机数误用 |
| DESER | 反序列化漏洞 | 不安全的数据解码 |
| RACE | 竞态条件 | 并发安全问题 |
| SSTI | 模板注入 | 服务端模板执行注入 |
| DOS | 拒绝服务 | 资源耗尽类攻击 |
| INFO | 信息泄露 | 敏感数据暴露 |
| FILE | 不安全文件操作 | 文件上传/权限问题 |
| GRPC | gRPC安全 | gRPC/Protobuf专属问题 |
| GOLNG | Go语言特有 | Go语言特性相关安全问题 |
| SUPPLY | 供应链攻击 | 依赖和构建安全 |
| REDIR | 开放重定向 | URL重定向操控 |
| LOG | 日志注入 | 日志内容操控 |
| XXE | XML外部实体 | XML处理安全 |

pattern_id格式示例：`GO-ATK-SQLI-001`、`GO-ATK-GOLNG-003`

## 阶段4：质量验证与去重

### 4.1 验证检查

对每个结构化的攻击模式条目执行验证：

1. **格式验证** — 必填字段是否齐全，字段类型是否正确
2. **代码验证** — 漏洞代码和安全代码是否为有效Go语法
3. **逻辑一致性** — 漏洞描述与代码示例是否匹配，安全模式是否真正修复了漏洞模式中的问题
4. **分类准确性** — CWE编号和严重性评级是否恰当
5. **可操作性** — 测试方法是否可复现，修复建议是否可实施

可使用验证脚本辅助检查：运行 `python scripts/validate_pattern.py <pattern_file>`

### 4.2 去重处理

在模式库积累过程中，不同来源可能产出重复或近似的模式。去重策略：

1. **精确匹配** — 相同 `category` + 相同 `sink_functions` + 相似 `code_example` → 合并为一条，保留最完整的信息
2. **近似匹配** — 相同 `category` + 不同变体 → 保留两条，添加 `related_patterns` 交叉引用
3. **互补匹配** — 不同来源描述同一模式的不同方面 → 合并字段，标注各字段来源

### 4.3 质量评级

对最终条目评定质量置信度：

| 置信度 | 条件 |
|--------|------|
| **高** | 全部必填和推荐字段已填充，代码经过验证，有多来源交叉验证 |
| **中** | 全部必填字段已填充，代码基本正确，单来源但信息完整 |
| **低** | 仅必填字段已填充，部分字段标记为"待补充"，需进一步验证 |

## 阶段5：库输出与索引

### 5.1 输出格式

攻击模式库输出为结构化Markdown文件，存储在 `vuln-lib/` 目录下：

```
vuln-lib/
├── index.md                    # 模式库索引与统计
├── patterns/                   # 按类别组织的模式文件
│   ├── sqli-patterns.md        # SQL注入模式集
│   ├── cmdi-patterns.md        # 命令注入模式集
│   ├── xss-patterns.md         # XSS模式集
│   ├── ssrf-patterns.md        # SSRF模式集
│   ├── auth-patterns.md        # 认证缺陷模式集
│   ├── golng-patterns.md       # Go语言特有模式集
│   └── ...                     # 其他类别
└── sources/                    # 来源追溯
    └── source-registry.md      # 来源注册表
```

### 5.2 索引文件结构

`index.md` 必须包含：

```markdown
# Go攻击模式库索引

## 库统计

| 指标 | 数值 |
|------|------|
| 模式总数 | [N] |
| 覆盖类别数 | [N] |
| 高置信度模式 | [N] |
| 中置信度模式 | [N] |
| 低置信度模式 | [N] |
| 来源报告数 | [N] |
| 最后更新 | [YYYY-MM-DD] |

## 类别分布

| 类别 | 模式数 | 文件 |
|------|--------|------|
| SQL注入 | [N] | [patterns/sqli-patterns.md](patterns/sqli-patterns.md) |
| ... | ... | ... |

## 严重性分布

| 严重性 | 模式数 |
|--------|--------|
| 严重 | [N] |
| 高危 | [N] |
| 中危 | [N] |
| 低危 | [N] |
```

### 5.3 模式文件格式

每个类别的模式文件（如 `sqli-patterns.md`）按以下格式组织：

```markdown
# [类别名称] 攻击模式集

## 模式列表

| ID | 名称 | 严重性 | 置信度 | 来源 |
|----|------|--------|--------|------|
| GO-ATK-XXX-001 | ... | ... | ... | ... |

---

## GO-ATK-XXX-001：[模式名称]

**严重性：** [严重/高危/中危/低危]
**CWE：** [CWE编号]
**置信度：** [高/中/低]
**来源：** [来源类型和引用]

### 漏洞描述

[背景、原理、影响]

### 前置条件

- [条件1]
- [条件2]

### 测试方法

**检测策略：** [策略描述]

**检测点：**
- [检测点1]
- [检测点2]

**测试步骤：**
1. [步骤1]
2. [步骤2]

**适用工具：** [工具列表]

### 漏洞模式（典型代码）

[完整Go代码示例，标注漏洞点]

### 数据流

[Source → Sink 路径描述]

### 安全模式

[修复后的Go代码示例]

**修复策略：** [修复思路]

**纵深防御：**
- [防御措施1]
- [防御措施2]

### 元数据

- Go版本：[适用版本]

- 标签：[标签列表]

```

### 5.4 来源注册

`source-registry.md` 记录所有输入来源的元数据：

```markdown
# 来源注册表

| 来源ID | 类型 | 文件路径 | 提取模式数 | 提取日期 |
|--------|------|----------|------------|----------|
| SRC-001 | vuln-insight | reports/xxx-insight.md | [N] | [日期] |
| SRC-002 | codehub-issue | reports/xxx-issues.md | [N] | [日期] |
| ... | ... | ... | ... | ... |
```

## 增量更新

当新的来源报告产生时，按以下流程增量更新模式库：

1. 从阶段1开始处理新来源
2. 在阶段4的去重步骤中，与现有库进行比对
3. 新模式追加到对应类别文件末尾
4. 对已有模式的补充信息合并到现有条目
5. 更新 `index.md` 的统计数据
6. 在 `source-registry.md` 中注册新来源

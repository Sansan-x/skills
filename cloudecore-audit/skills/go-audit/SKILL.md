# Go 代码安全审计

针对 Go 代码库的系统化多阶段安全审计能力（仅执行阶段4-7）。它必须读取上游 `project-analyzer` 输出的“审计策略计划”和“审计模式”，并严格遵守其中的范围/顺序/终止条件，禁止自行生成阶段1-3内容。

## 输入约束

阶段 4 开始前读取策略文件（如 `cat ./reports/audit-strategy-plan.md`）。若文件不存在或为空，中止并提示：`错误：未找到审计策略计划文件 ./reports/audit-strategy-plan.md。请先运行 project-analyzer（阶段1-3）生成该文件后再执行 go-audit。`

若文件存在：复述并核对是否包含以下项；**缺失项只能标「不确定」，不得臆想补全**：

- 漏洞类别优先级、信任边界→关键模块映射、模块审查顺序与 `stop_conditions`、重点 sink/source/净化检查点
- 审计模式（快速扫描 / 深度审计）

以该文件为阶段 4–7 的**唯一**依据。

## 阶段4：LLM 代码审计

核心步骤：漏洞发现（4.2）→ 数据流追踪（4.3，深度模式）→ 误报验证（4.4，深度模式）。4.1 为阶段 4 第一步，不可跳过。

### 4.1 加载漏洞模式库

**探测**（仅同级 `skills` 下的 go-vuln-lib）：

```bash
ls ../go-vuln-lib/vuln-lib/patterns/ 2>/dev/null
```

**主库选择**

- 若 `../go-vuln-lib/vuln-lib/patterns/` 存在且目录内有文件：主库为 **go-vuln-lib**；`ls -la` 该目录；按策略中的漏洞类别优先级按需读模式文件；记录来源 `go-vuln-lib` 与实际路径。
- 否则：主库为 **[references/vulnerability-patterns.md](./references/vulnerability-patterns.md)**，按策略类别定位章节按需加载；记录来源 `内置(references/vulnerability-patterns.md)`。

**共同加载（只写一次）**

- 必载 Go 扩展：[references/go-language-unique-patterns-extra.md](./references/go-language-unique-patterns-extra.md)（并发、生命周期、context、unsafe、reflect、cgo 等）。
- 策略含 **5GC 分支** 时加载：[references/5gc-protocol-vulnerability-patterns.md](./references/5gc-protocol-vulnerability-patterns.md)。

**加载结果**（写入报告附录方法论），逐项确认：

- 主库来源：`go-vuln-lib` / 内置
- 主库路径
- 已加载模式数 N、类别列表
- Go 扩展：`./references/go-language-unique-patterns-extra.md`，已加载数 M 或未加载
- 5GC 扩展：`./references/5gc-protocol-vulnerability-patterns.md`，已加载数 K 或未加载（非 5GC）

无论来源如何，每个漏洞模式应包含：
- 模式描述与风险等级
- 漏洞代码特征（查找目标）
- Sink函数和危险API
- 常见 source → sink 数据流
- 修复指南

### 4.2 漏洞发现

按输入审计策略计划确定的**模块/目录顺序**，系统性审查代码。不要把“少量具体文件”当作全部范围。

对每个模块/目录：

1. **模式匹配** — 将代码与漏洞模式库比对，查找：
   - 直接使用危险API（如SQL查询中的 `fmt.Sprintf`、带用户输入的 `os/exec`）
   - 信任边界处缺失输入校验
   - 不安全的默认配置
   - 泄露敏感信息的错误处理
   - 并发代码中的竞态条件
   - 密码学原语的不当使用
   - Go语言特有攻击面（unsafe包滥用、reflect注入、cgo边界问题等）

2. **语义分析** — 超越模式匹配，理解代码意图：
   - 输入校验是否真正有效，能否被绕过？
   - 授权检查是否覆盖所有相关路径？
   - 是否存在关于数据格式或信任级别的隐含假设？

3. **记录每个发现**，必须包含详细的漏洞代码片段：
   - 漏洞类型（来自模式库分类）
   - 位置（文件、函数、行范围）
   - Sink点（危险操作）
   - 疑似Source（不可信数据来源）
   - 初步严重性评估
   - **完整的漏洞代码片段**（包含上下文，不少于包含sink的完整函数体）

#### CoverageBackfill（覆盖补扫）

模块审查后必做：各模块标 `covered` / `uncovered`；`uncovered` 在预算内补扫为 `backfilled` 或保留；高优先级模块须全部被扫过。输出：

```
CoverageBackfill:
  total_modules: [N]
  covered: [A]
  backfilled: [B]
  uncovered: [C]
  coverage_rate: [(A+B)/N]
```

- **快速扫描**：进入阶段 5，跳过 4.3、4.4。
- **深度审计**：继续 4.3、4.4。

### 4.3 数据流追踪（污点分析）

对4.2中发现的每个漏洞，从 source 到 sink 追踪数据流以确认可利用性。此步骤采用“**工具优先，推断降级**”策略：

- 优先通过 Joern MCP 获取确定的调用链/污点路径
- 仅当工具失败或无结果时，才允许 LLM 生成推断链路
- 推断链路必须与工具链路分开输出，不得混写

#### 步骤A：输入标准化

为每个漏洞构造追踪输入：

1. `source_candidates[]`：不可信输入点（HTTP 参数、协议消息字段、Header、外部输入）
2. `sink_candidates[]`：危险操作点（SQL/exec/文件写入/协议状态写入/规则编程）
3. `target_files[]` / `target_functions[]`：漏洞所在文件与关联函数
4. `max_depth`：调用链追踪深度（避免无限展开）

#### 步骤B：调用 Joern MCP（通用契约）

在每个漏洞追踪时，优先调用 Joern 接口（无需绑定具体方法名，按通用请求/响应契约处理）。

请求字段（示例）：

```json
{
  "project_root": ".",
  "language": "go",
  "sources": ["..."],
  "sinks": ["..."],
  "target_files": ["..."],
  "target_functions": ["..."],
  "max_depth": 12,
  "timeout_ms": 120000
}
```

#### 步骤C：解析 Joern 响应

响应按以下结构解析（字段名允许映射，但语义必须一致）：

- `paths[]`：路径集合
- 每条 `path` 包含有序节点 `nodes[]`
- 每个节点至少含：`file`, `function`, `line`, `code`, `edge_type`

#### 步骤D：链路完整性校验（必须执行）

- 元信息：追踪编号、漏洞类型、`VULN-[ID]`
- **证据**：`ChainEvidenceType`（ToolConfirmed / LLMInferred）、`ToolName`、`ToolQuerySummary`、`ChainCompleteness`、`MissingSegments`、`ConfidenceCapReason`
- **Source**：位置、类型（HTTP/体/URL/Cookie/Header/外部等）、表达式
- **传播路径**：表列 序号 | 位置 | 操作 | 污点状态
- **Sink**：位置、危险 API、表达式
- **净化**：有无措施、详情、有效性（有效/无效/不适用）
- **结论**：可达性、可利用性、完整调用链文字

### 4.4 误报验证

对已做数据流追踪的漏洞，结合净化、可达性、可利用性做判定；**逐项依据**下表（五问、Go 模式、5GC 抑制），勿与表意重复展开长叙述。

#### 必答验证检查清单

| # | 检查项 | 问题 | Yes = 可能误报 |
|---|--------|------|----------------|
| 1 | **Source 可控性** | 该 source 是否真的可被外部攻击者控制？区分：外部 HTTP 参数 vs 内部配置 vs 硬编码常量 vs 协议消息中的强类型字段 | Source 不可被外部控制 → 排除 |
| 2 | **中间净化/校验** | source 到 sink 路径上是否存在有效净化？列出已识别的净化点并判定有效性 | 存在有效净化 → 排除 |
| 3 | **框架/ORM 自动防护** | 是否使用了框架级自动防护？（GORM 参数化、protobuf 强类型、html/template 自动转义等） | 有框架防护 → 排除 |
| 4 | **路径可达性** | 该代码路径是否真的可达？（是否为死代码、仅测试代码、需要特定 build tag、需管理员权限） | 路径不可达 → 排除 |
| 5 | **利用约束** | 即使可达，数据格式/类型约束是否使利用不可行？（如 protobuf int32 不会包含 SQL 注入载荷） | 格式约束阻断利用 → 排除 |

#### Go 框架/语言级常见安全模式识别表

| 安全模式 | 适用漏洞类型 | 判定 |
|----------|-------------|------|
| GORM/ent/sqlx 使用 `?` 占位符的参数化查询 | SQL 注入 | 排除（除非存在 `Raw`/`Exec` 拼接绕过） |
| `html/template` 自动 HTML 转义 | XSS | 排除（除非使用 `template.HTML()` 类型转换） |
| protobuf 强类型反序列化（字段是 int/enum/bytes，非任意字符串） | 注入类漏洞 | 排除（强类型字段不含注入载荷） |
| `net/http` 标准库 XML 解析 | XXE | 排除（Go encoding/xml 默认不处理外部实体） |
| `database/sql` 的 `Prepare`/`Query` 带占位符 | SQL 注入 | 排除 |
| Go `encoding/json` 反序列化到强类型结构体 | 反序列化/类型混淆 | 降低（vs 反序列化到 `interface{}` 需关注） |
| gRPC 拦截器链中已配置认证/授权中间件 | 认证绕过 | 排除（除非存在绕过拦截器的路径） |

#### 5GC 协议层误报抑制规则

| 场景 | 误报抑制规则 |
|------|-------------|
| 协议消息经过完整的 mandatory IE 校验和长度检查后 | 后续字段使用不再视为"未校验输入"（已通过协议层净化） |
| PFCP/NAS 消息来自已建立的安全关联（SCTP/TLS 认证） | 内部字段不等同于"外部用户输入"，但仍需 ownership 校验 |
| 3GPP 规范定义的枚举值/固定格式字段（Cause Value、Message Type 等） | 不是自由文本，不适用字符串注入类漏洞 → 排除 |
| NF 间 SBI 调用已通过 mTLS + OAuth2 token 双重认证 | SBI handler 不需要额外的"用户级"认证 → 排除认证缺失误报 |
| TEID/SEID 等标识符为固定长度数字类型 | 不适用字符串注入/路径穿越类漏洞 → 排除 |

#### 每个发现的判定

- **Confirmed**：路径清晰、无有效净化、未命中上表排除；且须 `ToolConfirmed` 或等效代码证据。
- **Likely**：路径存但依赖运行时条件。
- **Suspicious**：有显著缓解（在必答清单注明）。
- **False Positive**：命中必答排除、Go 表或 5GC 表；写明原因。

**硬规则**：`LLMInferred` 且无补充代码证据 → 不得 `Confirmed`；`ChainCompleteness` 为部分/断裂 → 默认 `Suspicious` 或 `False Positive`。移除误报；存疑项保留并附说明。

## 阶段5：漏洞分类与评级

对 Confirmed / Likely 漏洞映射 CWE 并按 CVSS 维度评级。

### 5.1 分类

将每个漏洞映射到CWE类别：

| 类别 | 常见CWE编号 |
|---|---|
| 注入 | CWE-89（SQL）、CWE-78（OS命令）、CWE-79（XSS）、CWE-917（表达式语言） |
| 认证缺陷 | CWE-287、CWE-384、CWE-613 |
| 敏感数据泄露 | CWE-200、CWE-312、CWE-319 |
| 访问控制缺陷 | CWE-862、CWE-863、CWE-639 |
| 安全配置错误 | CWE-16、CWE-1188 |
| 密码学失败 | CWE-326、CWE-327、CWE-330 |
| SSRF | CWE-918 |
| 路径穿越 | CWE-22 |
| 竞态条件 | CWE-362 |
| 反序列化 | CWE-502 |

### 5.2 严重性评级（CVSS 对齐）

| 因素 | 评估标准 |
|---|---|
| **攻击向量** | 网络 / 相邻 / 本地 / 物理 |
| **攻击复杂度** | 低（容易利用）/ 高（需要特定条件） |
| **所需权限** | 无 / 低 / 高 |
| **用户交互** | 无 / 需要 |
| **机密性影响** | 无 / 低 / 高 |
| **完整性影响** | 无 / 低 / 高 |
| **可用性影响** | 无 / 低 / 高 |

| 严重性 | CVSS分数范围 | 说明 |
|---|---|---|
| **致命（Critical）** | 9.0 – 10.0 | 远程利用，无需认证，高影响 |
| **严重（High）** | 7.0 – 8.9 | 显著影响，中等利用难度 |
| **一般（Medium）** | 4.0 – 6.9 | 有限影响或较高利用难度 |
| **提示（Low）** | 0.1 – 3.9 | 最小影响，难以利用 |

## 阶段6：攻击链组合分析

多漏洞可组合放大影响；链须**因果成立、端到端连续、有阶段 4 代码与数据流证据、现实可利用**，禁止臆造关联。

**过程**：建能力邻接图 → 找输出满足下一漏洞前置条件的组合（如泄露→注入、认证绕过→其他端点、SSRF→内网、路径穿越→密钥）→ 逐步验证 → 组合严重性看最终影响。

- **因果关系必须** — 链中每一步必须产出具体结果（数据、权限、状态变更），使下一步成为可能
- **连续性** — 链必须端到端可执行，无断裂；如果第N步的输出无法实际输入第N+1步，则链无效
- **不臆想** — 不得在无关漏洞间捏造关联；每个环节必须有阶段4中代码和数据流证据支撑
- **实际可利用** — 组合攻击必须代表现实场景，而非理论最坏情况

### 6.2 链构建过程

1. **构建漏洞邻接图** — 对每个发现，列出攻击者利用后获得的能力（如"读取任意文件"、"绕过端点X的认证"、"以数据库用户Y执行SQL查询"）
2. **识别链候选** — 寻找一个漏洞的输出能启用另一个漏洞前置条件的组合：
   - 信息泄露 → 启用定向注入（如泄露的数据库schema启用SQL注入）
   - 认证绕过 → 启用对含其他漏洞端点的访问
   - SSRF → 启用对安全性较弱的内部服务的访问
   - 路径穿越 → 读取配置/密钥 → 权限提升
3. **验证每条链** — 逐步走查，确认每个过渡有实际代码路径和数据流支撑
4. **评估组合影响** — 链的严重性基于最终可达影响，而非各漏洞严重性的简单叠加

### 6.3 输出格式

对每条有效攻击链：

```
攻击链: [描述性名称]
组合严重性: [严重/高危/中危]
步骤:
  1. [漏洞A] → 攻击者获得 [具体能力]
  2. [漏洞B]（由步骤1启用）→ 攻击者获得 [升级能力]
  3. ...
最终影响: [攻击者最终达成的目标]
前置条件: [攻击者在步骤1前需要的条件]
```

如果未发现有效攻击链，明确说明——不要强行组合不成立的组合。

## 阶段7：审计报告生成

生成全面的中文安全审计报告。加载报告模板：[references/report-template.md](./references/report-template.md)

### 报告输出

报告输出到 `./reports/` 目录下，文件命名规则：

```
./reports/[项目名称]-goaudit-[YYYYMMDD-HHMM].md
```

例如：`./reports/myproject-goaudit-20260319-1430.md`

在生成报告前，先创建 `./reports/` 目录：

```bash
mkdir -p "$(pwd)/reports"
```

项目名称从 `go.mod` 的 module 声明中提取（取最后一个路径段），如果不可用则使用项目根目录名。时间使用审计执行时的当前时间。

### 报告结构

报告必须包含：

1. **执行摘要** — 高层发现、整体风险态势、关键数字（致命/严重/一般/提示计数）
2. **项目概况** — 来自 project-analyzer 输出的项目背景（类型、技术栈、架构）
3. **审计范围与方法** — 审计了什么、使用的模式和方式、运行的工具（如有）、覆盖的文件和模块
4. **发现汇总表** — 所有发现按严重性排序，含ID、标题、严重性、CWE、位置
5. **详细发现** — **发现汇总表中的所有漏洞必须全部输出在此章节中**，按等级排列（致命 → 严重 → 一般 → 提示），每个漏洞包含完整的6个必填子章节（漏洞描述、漏洞代码、数据流路径、利用场景含PoC、影响、修复建议含前后对比）
6. **攻击链分析** — 来自阶段6（如有链被识别）
7. **修复优先级矩阵** — 按影响/投入比排序的修复清单
8. **附录**

### 第3章显示约束

- 第3章“审计范围与方法”不得展示“模式库来源”相关信息（包括来源类型、模式库路径、已加载模式数、已加载模式类别）。
- 模式库来源信息仅允许出现在附录的方法论说明中。

### 详细发现全量输出要求

所有漏洞必须全部输出在主报告的第5章"详细发现"中，不使用分文件方式。这是确保审计人员能在一份报告中看到所有安全问题的核心要求。

**篇幅受限时的处理方式：** 如果漏洞数量较多，按以下策略分批输出，但最终所有漏洞都必须出现在同一份报告中：

1. **按等级分批生成** — 先输出致命级全部漏洞，再输出严重级全部漏洞，依次输出一般级和提示级。每批输出后继续下一批，直到所有漏洞输出完毕
2. **如果单次输出被截断** — 在后续的继续输出中接续上次中断的位置，将剩余漏洞追加到报告中，确保最终报告完整
3. **禁止省略或简化**  — 不得以任何理由跳过漏洞、合并漏洞或简化后续漏洞的子章节内容
4. **checklist** — 检查详细发现中的漏洞数量与发现汇总漏洞数量一致才算完成任务

### 报告质量要求

- **漏洞代码：** 每个发现必须包含完整的漏洞代码片段（不少于包含sink的完整函数体），包括文件路径和行号，使用 `// ↑ 漏洞:` 注释标注漏洞行
- **数据流路径：** 每个发现必须包含结构化数据流路径（使用阶段4.3定义的表格格式），明确标注 Source、传播路径中的每个节点、Sink、净化检查结果和结论
- **利用PoC：** 每个发现的利用场景必须包含可直接复现的PoC代码或命令，而非仅文字描述
- **修复对比：** 每个发现的修复建议必须展示修复前（漏洞代码）和修复后（安全代码）的完整代码对比
- 严重性评级必须有理由支撑，不能仅仅赋值
- 报告应可操作——开发者仅凭报告即可修复每个问题

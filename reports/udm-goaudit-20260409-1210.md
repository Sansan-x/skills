## 详细发现

### VULN-003: reflect.DeepEqual时序攻击 (严重)

**类别**: GENERAL_GO_SECURITY_TRAPS
**状态**: CONFIRMED
**CPG验证**: ✅ 已验证数据流路径

#### 漏洞描述

在5G AKA认证重同步流程中，使用`reflect.DeepEqual`比较MAC-S值，而非恒定时间比较函数。攻击者可通过测量比较时间推断MAC值差异，实现时序攻击。

#### CPG验证的数据流路径

```
Source: api_ueauthentication.go:127
  └─ c.GetRawData() → requestBody

Step 2: api_ueauthentication.go:141
  └─ openapi.Deserialize(&authInfoReq, requestBody)

Step 3: api_ueauthentication.go:180
  └─ GenerateAuthDataProcedure(c, authInfoReq, supiOrSuci)

Step 4: generate_auth_data.go:294
  └─ hex.DecodeString(authInfoRequest.ResynchronizationInfo.Auts) → Auts

Step 5: generate_auth_data.go:322
  └─ p.aucSQN(opc, k, Auts, randHex) → macS

Sink: generate_auth_data.go:323
  └─ reflect.DeepEqual(macS, Auts[6:])
```

#### 受影响代码

```go
// internal/sbi/processor/generate_auth_data.go:323
if reflect.DeepEqual(macS, Auts[6:]) {
    // MAC验证通过
}
```

#### 修复建议

```go
// 使用恒定时间比较
import "crypto/hmac"

if hmac.Equal(macS, Auts[6:]) {
    // MAC验证通过
}
```

#### 攻击场景

1. 攻击者发送带有错误MAC-S的认证重同步请求
2. 测量服务器响应时间
3. 逐字节推断正确的MAC-S值
4. 绕过认证验证

### VULN-005: HTTP请求体无大小限制 (一般)

**类别**: DOS_RESOURCE_LIMITS
**状态**: CONFIRMED
**CPG验证**: ✅ 已验证15处调用点

#### 漏洞描述

所有SBI API处理器使用`c.GetRawData()`读取请求体，但未设置大小限制。攻击者可发送超大请求体耗尽服务器内存。

#### CPG验证的受影响位置

| 文件 | 行号 | 方法 |
|------|------|------|
| api_eventexposure.go | 50 | HandleCreateEventExposureSubsc |
| api_eventexposure.go | 95 | HandleModifyEventExposureSubsc |
| api_httpcallback.go | 35 | HandleAmfStatusChangeNotify |
| api_parameterprovision.go | 113 | HandleModifyParameterProvision |
| api_subscriberdatamanagement.go | 199 | HandleCreateSdmSubscriptions |
| api_subscriberdatamanagement.go | 236 | HandleModifySdmSubscriptions |
| api_subscriberdatamanagement.go | 325 | HandleCreateSharedDataSubscr |
| api_subscriberdatamanagement.go | 361 | HandleModifySharedDataSubscr |
| api_ueauthentication.go | 46 | HandleConfirmAuth |
| api_ueauthentication.go | 127 | HandleGenerateAuthData |
| api_uecontextmanagement.go | 293 | HandleRegisterAmf3gppAccess |
| api_uecontextmanagement.go | 376 | HandleDeregisterAmf3gppAccess |
| api_uecontextmanagement.go | 460 | HandleRegisterSmfNon3gppAccess |
| api_uecontextmanagement.go | 528 | HandleDeregisterSmfNon3gppAccess |
| api_uecontextmanagement.go | 664 | HandleSmfDeregistrationInAmf |

#### 修复建议

```go
// 在gin引擎配置中设置最大请求体大小
router := gin.Default()
router.MaxRequestSize = 10 * 1024 * 1024 // 10MB

// 或在中间件中限制
func LimitRequestBody(maxSize int64) gin.HandlerFunc {
    return func(c *gin.Context) {
        c.Request.Body = http.MaxBytesReader(c.Writer, c.Request.Body, maxSize)
        c.Next()
    }
}
```

### VULN-004: 密钥材料日志泄露 (一般)

**类别**: SENSITIVE_ASSET_LEAKAGE
**状态**: SUSPECTED
**CPG验证**: ✅ 已验证8处日志点

#### 漏洞描述

认证数据处理流程中使用Trace级别日志输出密钥材料。虽然Trace级别默认关闭，但生产环境配置错误时可能导致敏感信息泄露。

#### CPG验证的日志泄露点

| 行号 | 日志内容 |
|------|----------|
| 50 | `SQNms=[%x]` |
| 54 | `macS=[%x]` |
| 255 | `K=[%x], sqn=[%x], OP=[%x], OPC=[%x]` |
| 288 | `RAND=[%x], AMF=[%x]` |
| 425 | `AUTN=[%x]` |
| 447 | `xresStar=[%x]` |
| 457 | `Kausf=[%x]` |
| 484 | `ckPrime=[%x], kPrime=[%x]` |

#### 修复建议

```go
// 移除敏感数据的完整输出
// 原代码:
logger.UeauLog.Tracef("K=[%x], sqn=[%x]", k, sqn)

// 修复后:
logger.UeauLog.Tracef("K present=%v, sqn len=%d", k != nil, len(sqn))
// 或完全移除敏感日志
```

### VULN-007: Goroutine无超时控制 (提示)

**类别**: CONCURRENCY_LIFECYCLE
**状态**: SUSPECTED
**CPG验证**: ⚠️ gosrc2cpg无法解析GoStmt

#### 漏洞描述

HTTP回调通知在独立goroutine中执行，但未设置超时控制。目标服务无响应时可能导致goroutine累积。

#### 修复建议

```go
go func() {
    ctx, cancel := context.WithTimeout(context.Background(), 30*time.Second)
    defer cancel()
    p.SendOnDeregistrationNotificationWithContext(ctx, ueID, deregCallbackUri, deregistData)
}()
```
